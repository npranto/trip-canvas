# TC-005 — Define app routes and major components

**Status:** ✅  
**Owner:** You + Junior  
**Goal:** Map pages and reusable UI structure.

**Stack (per `PLAN.md`, confirmed):** Next.js (App Router), TypeScript, Tailwind CSS.

**Routing conventions (confirmed):** Trip intake lives on a **dedicated page** (not the landing URL). Each plan has a **stable, shareable** canonical URL under **`/trip/[tripId]`** after creation. Use a static segment **`/trip/new`** for the intake form so `[tripId]` never equals `new`.

### component inventory defined

Grouped by product area from `PLAN.md` §1 and downstream tickets; names are **logical**—rename to match chosen design system.

**App shell & global**

- Root layout wrapper; optional header / footer; global loading and error affordances; trust / AI-disclosure messaging where appropriate (§1.1, §7 principles).

**Landing**

- Hero and value proposition; example prompts as **teasers** linking to **`/trip/new`**; primary CTA to **`/trip/new`**; secondary CTA to **`/trips`** when persistence exists (§1.1).

**Trip request**

- Natural-language input; optional structured fields (destination, duration, dates, traveler type, budget, interests); validation and hints; submit and loading states; sanitization/normalization hooks (§1.2).

**Generation & system feedback**

- Latency/loading experience; retry; error and fallback copy tied to incomplete AI output (§1.3).

**Trip results (maps to TC-051–TC-054)**

- Trip summary block (title, destination, duration, budget label, summary text).
- Assumptions + confidence / uncertainty / fallback presentation.
- Itinerary: day cards, activities, food/area suggestions, per-day metadata, empty-day handling.
- Hotel and flight recommendation cards (including placeholders, reasons, future CTA hooks).

**Refinement**

- Quick actions (cheaper, fewer activities, more family-friendly, closer to center, etc.); freeform refinement input; loading while regenerating; version/history presentation (§1.5, TC-061, TC-062).

**Save, list, manage**

- Save trip action and feedback; saved trips list with metadata preview; trip detail; editable assumptions UI; delete/archive/duplicate with confirmations (§1.6, TC-071–TC-074).

**Future / phase-gated (still name in inventory for extensibility)**

- Auth UI; profile/preferences form; share/copy link UI (TC-111); booking handoff on cards (TC-094); paywall/upgrade surfaces (TC-103).

**Cross-cutting UI**

- Reusable primitives (buttons, inputs, cards, modals) per design system (PLAN “component-driven architecture”, TC-021); accessible, responsive patterns (PLAN principles, TC-130).

### shared layout rules defined

- **Next.js App Router:** use **`app/layout.tsx`** (root) plus route-group or nested layouts only where they reduce duplication (e.g. optional marketing vs “app” chrome)—keep depth shallow for MVP.
- **Productized layout, not chat-first:** structured sections for summary, assumptions, itinerary, recommendations, and confidence—not a single transcript stream (intro + §1.4, master context in PLAN).
- **Consistent hierarchy:** clear headings and grouping so scanning matches the domain model (trip → days → items → recs).
- **Responsive baseline:** landing, **`/trip/new`**, and **`/trip/[tripId]`** usable on mobile; dense data (itinerary, cards) reflows safely (§1.1, §1.4, Tier 2 “polished responsive experience”).
- **State visibility:** loading, error, empty, and partial/fallback content are first-class in layout (spacing, placement), not afterthoughts (§1.3, §1.4, engineering principles).
- **Shareable trip URL:** anything shown on **`/trip/[tripId]`** should assume **bookmarking and sharing**; avoid trip state that exists only in transient client memory without a server or URL backing (TC-050).
- **Extensibility:** shell and section boundaries allow adding auth-gated areas, saved trips, and share views without rewriting core results layout (Phase 1–3 split in PLAN).
- **Implementation note:** concrete shell (root layout, container/max-width, spacing tokens, header/footer) is built in **TC-020**; this ticket defines the **rules** those implementations must follow.

### page responsibilities documented

| Route | Responsibility |
| --- | --- |
| **`/`** | Explain product; AI/trust clarity; drive users to **`/trip/new`**; optional link to **`/trips`**; no primary trip form on this URL (§1.1). |
| **`/trip/new`** | Turn vague or detailed NL into a submittable request; optional structured fields; validation; normalization; trigger generation; then land user on **`/trip/[tripId]`** (§1.2, §1.3). |
| **`/trip/[tripId]`** | Single canonical surface for generated and saved plans: summary, assumptions, budget, itinerary, hotel/flight blocks, rationale/confidence, refinement (§1.4–§1.5), save/share, and post-save management (TC-071–TC-074) as features land. |
| **`/trips`** | List persisted trips with metadata; empty state; each row links to **`/trip/[tripId]`** (§1.6, TC-072). |
| **`/login`**, **`/profile`**, share variant, **paywall** (later) | As in Tier 2–3 and TC-080+, TC-083, TC-110, TC-103—same trip schema on **`/trip/[tripId]`**; access may be gated by auth or share policy. |
