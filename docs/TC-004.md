## Acceptance Criteria

### Documented types/schemas

#### Trip request

- **Required:** natural-language **prompt** (primary input).
- **Optional structured helpers** (align naming in types with UI): destination, trip duration, date/month, traveler type, budget, interests — per PLAN §1.2 / MVP §1.2.
- Document **normalization / sanitization** expectations before the generation API (PLAN §1.2, engineering principles in PLAN).

#### Trip assumptions

- Represents **inferred or user-edited** facts the plan depends on (PLAN §1.4 “assumptions used”, §4.3 “Editable assumptions”).
- Shape supports **edit and re-run**: each assumption has a **stable `id` or key** so the UI can update one item unambiguously (MVP §1.4 / §1.6).

#### Itinerary day

- **Day identity:** day index and/or calendar date (document which is canonical for sort/display).
- **Activities:** ordered list (title required; time, duration, description optional — document optional fields explicitly in types).
- **Food / area suggestions** (backlog TC-053, PLAN §1.4).
- **Per-day metadata:** optional bag or named fields, additive-only preference for future fields.

#### Recommendation item

- **Discriminated union:** `kind: "hotel" | "flight"`.
- **Hotel** (`PLAN.md` §3.1): name, neighborhood, price range, walkability or relevance score/label, reason recommended, image URL (placeholder allowed), external / booking URL.
- **Flight** (`PLAN.md` §3.2): origin, destination, approximate price, layover count, airline, reason recommended, link out.
- Schema must allow **MVP placeholders** and partial fields without breaking consumers (PLAN §1.4, TC-054 intent).

#### Trip plan response

- **Summary:** title, destination, duration, budget interpretation (PLAN §1.4, MVP §1.4).
- **Assumptions:** trip-assumptions shape above.
- **Itinerary:** array of itinerary-day objects.
- **Recommendations:** hotel and flight lists (may be empty or placeholder-heavy in MVP).
- **Rationale / explanation** and **confidence / uncertainty:** document in types whether these are **structured** (e.g. labels + text) or plain strings so results UI stays consistent.
- **Identity:** stable **trip/plan id** for results routing and persistence (PLAN TC-050 “trip id state strategy”; MVP §1.6 metadata).
- **`schemaVersion`:** required on this envelope (see Contract standards).

#### Refinement request (minimum)

- **Plan reference:** identifier for the **current plan/trip** being refined (how the server resolves full context is extended in TC-060).
- **Intent:** either a **quick-action** enum — `cheaper`, `fewerActivities`, `moreFamilyFriendly`, `closerToCityCenter` — or **freeform text** (PLAN §1.5, MVP §1.5).
- **Optional scope hints** for **partial regeneration** (PLAN §1.5); v1 may still “replace full plan” server-side if documented — TC-060 refines without removing these fields.

---

### Versioned response shape agreed upon

- **Body:** all successful generation/refinement responses that carry a plan use the same **trip plan response** envelope including **`schemaVersion`**.
- **HTTP:** new externally stable APIs live under **`/api/v1/`** (or documented equivalent); breaking HTTP or payload changes require a new major URL segment and/or coordinated `schemaVersion` bump per team policy.
- **Refinement:** response uses the **same envelope** as initial generation; optional **revision / history** fields are introduced in **TC-060** and remain compatible with top-level versioning.

---

### Validation strategy defined

- **Server-side validation** immediately after structured model output, **before** response reaches the client (PLAN §4.1, TC-043, TC-003).
- **TC-004** defines **fields, optionality, and invariants**; **TC-043** implements parse, validate, normalize, and malformed handling.
- **Outcomes to document:** success; **degraded / partial** with safe empty sections or fallbacks; **recoverable client error** (bad input); **provider / model failure** with traceable error shapes (aligned with TC-042).
- **Observability:** failed validation is a first-class log/monitor signal (PLAN §6.2).
- **Client:** treats API responses as **authoritative** once shipped from the server; optional client-side guards are non-authoritative.

## Testing

- **Jest** (per TC-003): golden JSON fixtures per entity — valid minimal trip, full trip, degraded partial, invalid payloads.
- Tests exercise **parsing/validation** against `schemaVersion` and regress when the contract changes.
- Deeper **malformed LLM output** and normalization coverage belongs to **TC-043**; this ticket only requires that **basic schema tests are planned** and fixtures listed or stubbed in the same doc/repo area as types.
