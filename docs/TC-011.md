# TC-011 — Establish folder architecture and conventions

**Status:** ✅  
**Owner:** You  
**Goal:** Prevent random structure during sprint.

---

## Acceptance Criteria

- folders defined for:
  - app/routes
  - components
  - features
  - lib
  - services
  - types/schemas
  - tests

- naming rules documented
- feature ownership boundaries documented

## Conventions

**Naming**

- **Routes:** `app/<segment>/page.tsx` for pages; path strings live in `app/routes/` (e.g. `paths.ts`), not scattered as string literals.
- **Components:** `PascalCase.tsx` in `components/` for shared UI; prefix with domain when it helps (`TripCard`).
- **Features:** one folder per feature under `features/<kebab-name>/`; entry components `PascalCase.tsx`.
- **Lib:** `camelCase.ts` for pure helpers; no React components, no network or filesystem.
- **Services:** `camelCase.ts`, async when IO; return types from `types/schemas/` where shared.
- **Types:** shared shapes in `types/schemas/`; name files after the concept (`trip-preview.ts`).
- **Tests:** mirror source layout under `tests/` (e.g. `lib/foo.ts` → `tests/lib/foo.test.ts`).

**Feature ownership**

- A feature folder owns its UI and feature-local hooks/utilities. Prefer importing from `components/`, `lib/`, `services/`, and `types/schemas/` — avoid reaching into another feature’s internals (no `features/b` importing `features/a/some-internal.tsx`). Promote shared pieces to `components/` or `lib/` when two features need the same thing.
