# TripCanvas Design System

## 1. Design Direction

Based on the mockup, the product style should feel:

- calm
- modern
- trustworthy
- structured
- slightly editorial
- AI-native but not “sci-fi”
- product-first, not chatbot-first

The visual language suggests:

- soft blue brand tones
- airy whitespace
- rounded cards
- high legibility
- clear section separation
- subtle travel inspiration without clutter
- strong CTA emphasis
- content organized in cards, lists, chips, and timelines

---

# 2. Design Tokens

## 2.1 Color Tokens

Use semantic tokens instead of hardcoding colors into components.

### Brand

- `brand.primary` → main CTA blue
- `brand.primaryHover`
- `brand.primarySoft` → light blue backgrounds
- `brand.secondary` → muted navy / slate-blue for headings
- `brand.accent` → optional sky/cyan accent

### Neutrals

- `bg.canvas` → app background
- `bg.surface` → main white card background
- `bg.subtle` → soft panel background
- `border.default`
- `border.strong`
- `text.primary`
- `text.secondary`
- `text.muted`
- `text.inverse`

### Feedback

- `success.default`
- `warning.default`
- `error.default`
- `info.default`

### AI / Confidence

- `confidence.high`
- `confidence.medium`
- `confidence.low`
- `fallback.bg`
- `fallback.border`

### Suggested Tailwind mapping

You can map these into Tailwind theme extensions like:

- `tc-blue-600`
- `tc-blue-500`
- `tc-blue-100`
- `tc-slate-900`
- `tc-slate-700`
- `tc-slate-500`
- `tc-slate-200`
- `tc-slate-100`
- `tc-success`
- `tc-warning`
- `tc-danger`

---

## 2.2 Typography Tokens

### Font roles

- `font.display` → hero / page-level marketing headline
- `font.heading` → section/page headings
- `font.body` → body copy
- `font.label` → field labels, pills, metadata
- `font.mono` → optional for ids/debug metadata only

### Type scale

Suggested naming:

- `text-display-xl`
- `text-display-lg`
- `text-h1`
- `text-h2`
- `text-h3`
- `text-body-lg`
- `text-body`
- `text-body-sm`
- `text-label`
- `text-caption`

### Usage

- Landing hero title: `display`
- Page title: `h1`
- Section headings: `h2`
- Card titles: `h3`
- Body content: `body`
- Metadata/chips/helper text: `body-sm` / `label`

---

## 2.3 Spacing Tokens

Use an 8px system.

Suggested scale:

- `space-1` = 4
- `space-2` = 8
- `space-3` = 12
- `space-4` = 16
- `space-5` = 20
- `space-6` = 24
- `space-8` = 32
- `space-10` = 40
- `space-12` = 48
- `space-16` = 64

Use these consistently for:

- card padding
- vertical page rhythm
- form field spacing
- section spacing
- timeline gaps

---

## 2.4 Radius Tokens

The mockup leans soft and friendly.

- `radius.sm` → inputs, chips
- `radius.md` → buttons
- `radius.lg` → cards
- `radius.xl` → hero panels / featured modules
- `radius.full` → avatars / badges / loading dots

---

## 2.5 Shadow Tokens

Keep subtle.

- `shadow.xs` → lightweight borders
- `shadow.sm` → cards
- `shadow.md` → elevated panels / modals
- `shadow.focus` → focus ring style, not literal box shadow only

---

## 2.6 Motion Tokens

Motion should be calm and functional.

- `duration.fast`
- `duration.normal`
- `duration.slow`
- `ease.standard`
- `ease.emphasized`

Use motion for:

- button hover
- input focus
- skeleton shimmer
- card appearance
- loading transitions
- collapsible sections

Respect reduced motion.

# 8. Naming Strategy for Reusability

Use this layering:

### Level 1 — primitive

Generic and reusable anywhere

- `Button`
- `Card`
- `Input`

### Level 2 — pattern

Reusable inside product domain

- `SectionBlock`
- `TripMetaRow`
- `RefinementActionChip`

### Level 3 — feature component

Specific to business domain

- `TripSummaryCard`
- `HotelRecommendationCard`
- `SavedTripCard`

That structure keeps things scalable and avoids turning every element into a one-off custom component.

---

# 9. Component API Guidance

When you build these, keep the props stable and predictable.

Example mindset:

### Good

- `variant`
- `size`
- `title`
- `description`
- `actions`
- `status`
- `isLoading`

### Avoid

- too many boolean flags like:
  - `isPrimary`
  - `isSmall`
  - `isBlue`
  - `isTripMode`
  - `isSpecial`

Prefer enums/variants instead.

# 14. Strong Recommendation for Your Next.js Project

Use this rule:

**Primitives stay dumb. Feature components hold presentation logic. Pages orchestrate data.**

That means:

- `Button` should not know about trips
- `TripSummaryCard` can know about trip summary shape
- `page.tsx` should assemble feature components and fetch data

That separation will help a lot once AI generation, saved trips, and refinement history get more complex.
